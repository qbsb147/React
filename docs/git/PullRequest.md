# Pull Request Policy

## 1. PR는 작게 만듭니다.
- 변경 범위가 작을수록 리뷰가 빠르게 진행될 수 있습니다.

## 2. PR요청 전에 셀프 리뷰를 수행합니다.
- 기본적인 실수는 스스로가 잡도록 노력합니다.
- 검토자가 떠넘기는 식의 행동은 하지 않도록 합니다.
- 셀프 리뷰를 통해 코딩 사고 과정을 기릅니다.

## 3. 기능 PR과 포맷팅(코드 개선) PR는 분리합니다.
- 포맷팅 하고 기능 구현이 같이 있으면 포맷팅 때문에 기능 변경을 놓칠 수도 있습니다. 따라서 가능한 기능 구현할 때는 기능 구현만 하고 PR를 올리고 포맷팅 PR는 따로 올리도록 합니다.

## 4. Title은 의미 있게 짧고 명확하게 작성될 수 있도록 합니다.

## 5. PR에 '왜 이렇게 했는지'까지 작성합니다.
- 만약 선택사항이 여러 개 있을 때, 해당 방식을 고수했다면 그렇게 한 이유를 작성합니다.
- 어떠한 고민을 했는지를 명시함으로써 해당 코드에 대해 다른 팀원들의 이해도를 높일 수 있습니다.
- 사진 한 장이 변화를 시각적으로 이해하는데 큰 도움이 됩니다. 필요하다면 팀원들의 이해도를 높일 수 있는 사진을 첨부해줄 수 있도록 합니다.

## 6. 모든 코드를 보고 코멘트에 신경 써줍니다.
- 코드를 확인했으면 해당 내용을 확인했다는 것을 알릴 수 있는 간단한 이모지라도 달아줄 수 있도록 합니다.
- 해당 코드에 대한 코멘트를 달아주면 더욱 좋습니다.

## 7. 다른 팀원이 승인할 때까지 Merge를 하지 않도록 합니다.

## 8. 2개의 프로젝트 클론 생성
- 하나는 정상적인 업무용, 하나는 PR 검토용으로 프로젝트를 관리하는 것을 추천합니다.

## 9. 로컬에서 검증하기
- 개선할 수 있는 사항이 발견되면 코멘트를 하기 전에 로컬 클론을 변경해보고 검증해봅니다.
- 만약 제안이 꽤 커졌다면 대규모 제안을 PR로 생성해보도록 합니다.

## 10. 코멘트 달기
- 구체적인 대안이 없다면 코멘트를 하지 않도록 합니다.
- 잘 모르겠으면 몇가지 테스트를 한 다음 근거를 기반으로 코멘트를 작성합니다. 자신의 생각으로만 코멘트를 단다면 이는 해당 일을 남에게 넘기는 것입니다.
- 뉘앙스를 좋게 유지합니다. PR를 작성하는 것은 기본적으로 코멘트의 비판을 불러옵니다.
   - 이모티콘을 통해 분위기를 살릴 수 있습니다.
   - '너'보다는 '우리'라는 표현을 쓰도록 합니다.
   - 아니면 차라리 수동적 표현을 쓸 수 있도록 합니다.

## 11. 긴급 PR은 피합니다.
- 만약 긴급하게 Merge를 해야한다면 지금 당장 검토하라는 메세지를 보내지 않고 Merge를 수행할 수 있도록 합니다. [현 프로젝트에서는 해당 일은 없을 것 같지만...]
